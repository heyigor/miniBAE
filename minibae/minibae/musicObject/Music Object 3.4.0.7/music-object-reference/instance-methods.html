Embedding

MusicObject 2.0 introduced a new concept in Beatnik authoring and addressed many of the issues with cross-browser authoring. One of the significant changes in interactive authoring for Beatnik is that the author no longer embeds players into an HTML page with explicit HTML code, but instead uses the magicEmbed, stubEmbed, and preloadEmbed methods of the Music Object instance. The embedding methods will write the appropriate HTML into the page, depending on what browser the user has. In the case of Netscape Navigator, the embedding methods would write the HTML for a plug-in EMBED tag. And for Microsoft's Internet Explorer, the embedding methods would write the HTML for an ActiveX control OBJECT tag. The embedding methods also perform other useful steps needed in order to link a player instance with a Music Object instance.

 magicEmbed

 
 CATEGORY   Instance Method 
 MODULE   Music Object Core 
 Music Object Component - Basic

 File Types Supported   RMF, MIDI, WAV, AIFF, AU 
 

The magicEmbed method is the standard way to embed a player instance in an HTML page, so that the player instance can be controlled using the Music Object API.

    magicEmbed ( EmbedLineSTR )

Using this method is different to a pure HTML approach and requires you to create a SCRIPT block at the position in the HTML document where you would like to embed the player. You then enclose the call to the magicEmbed method within this SCRIPT block. The magicEmbed method takes care of some of the nasty little details for the author, such as the connection between the player instance and its associated Music Object instance. magicEmbed also sets up logical defaults for certain of the player's attributes so the author does not need to specify these. The magicEmbed method takes a string as a parameter - this string representing the EMBED tag that you would normally place in the HTML if you were only authoring for non-interactive music. In this string, it is also not necessary to specify the "<EMBED " text that opens the EMBED tag, nor is it necessary to use ">" to close the EMBED tag. For example, consider the following before-and-after example:

BEFORE (Using Standard HTML)
<EMBED
   SRC="my-file.rmf"
   TYPE="audio/rmf"
   WIDTH=144
   HEIGHT=60
   PLUGINSPAGE="http://www.beatnik.com/to/?player"
>
AFTER (Using the magicEmbed Method)
<SCRIPT LANGUAGE=JavaScript><!-- //

myMusicObject.magicEmbed ('SRC="my-file.rmf" WIDTH=144 HEIGHT=60');

// --></SCRIPT>

Some things to note
  • In the 'AFTER' example shown above, it is assumed that the author has already created the Music Object instance called 'myMusicObject'. The full template is explained in greater detail in the section Starting to use Music Object.
  • Notice how, in the magicEmbed method, single quotes are used to enclose the EMBED tag string. This is because the EMBED tag itself may contain double quotes that enclose some of the attribute values, such as the SRC attribute.
  • Notice how the TYPE and PLUGINSPAGE attributes are no longer in the EMBED tag that is passed to magicEmbed. This is because the magicEmbed sets up appropriate defaults for these attributes and it is not necessary for the author to specify them.

Authors that are coming from a background of developing for Internet Explorer and using ActiveX controls may find it curious to use the EMBED tag syntax to create a player instance that may end up being an ActiveX control if the user is browsing with IE. The rationale behind this approach is that many early Beatnik authors have been authoring for the Beatnik Plug-in for Netscape, simply because there was no ActiveX control available initially. Much of the Beatnik content is therefore still authored for a plug-in and the magicEmbed method offers an easy upgrade path for early developers because all they have to do is move their HTML EMBED tags into a JavaScript magicEmbed method. It is also beneficial to put forward only one embedding syntax for Beatnik so that authors do not need to be concerned with browser issues.

The magicEmbed method performs the following useful steps
  • automatically sets "audio/rmf" as the default value for the TYPE attribute. The user can still override this by specifying a value for the TYPE attribute, but when the TYPE is set to "audio/rmf" the author can always ensure that the browser will use Beatnik to handle file types that may otherwise be handled by another plug-in (such as QuickTime handling MIDI, or Yamaha MIDplug handling MIDI, or LiveAudio handling WAV files, etc.).
  • automatically sets the PLUGINSPAGE attribute's value so that users of Netscape Navigator who do not have the Beatnik Plug-in installed will go to the download page on the Beatnik Web site.
  • automatically sets the ONREADY attribute's value, specifying a callback function that links the player instance with the Music Object instance. Authors no longer need to be concerned with the mechanics behind ONREADY.
  • automatically sets the NAME attribute's value, specifying a name that links the player instance with the Music Object instance. Authors no longer need to understand the naming scheme that connects a player instance with its associated Music Object instance.
  • automatically sets the default values for the WIDTH and HEIGHT attributes. Whereas, before it could lead to problems if the author did not specify a value for WIDTH or HEIGHT, magicEmbed now sets 144 as the default WIDTH and sets 60 as the default HEIGHT. These defaults are still overridden by what the author specifies.

With the new mechanism for embedding truly hidden player instances introduced in Music Object version 3.3, a problem was found with embedding hidden instances in table cells, that would cause intermittent crashes in some 4.x versions of Netscape Navigator. Hidden instances should, therefore, not be embedded within table cells (or any other container elements, for that matter). Hidden instances should only be embedded as top level members of the BODY tag. For more information, consult the section Authoring Issues: Embedding Hidden Instances in Table Cells.


  • It is only possible to embed one player instance per Music Object instance.
  • Refer to the Embedding Properties reference for a full list of attributes that can be specified in the EMBED tag.
  • This method may only be called before the player instance has been embedded.

The preloadEmbed method lets you preload a music file into a hidden player instance, for the purpose of triggering the file to play at a later point.

    preloadEmbed ( FileUrlSTR )

The preloadEmbed method is similar to stubEmbed in that it is also a convenient short form for embedding a player. It is specifically geared towards the preloading of RMF or other files that are to be triggered as one-shots when a certain user interaction occurs in a page. The preloadEmbed method sets the WIDTH and HEIGHT attributes to 2 and uses the HIDDEN attribute to make the player invisible on the page. It also sets the AUTOSTART attribute to FALSE so that the file doesn't begin playing immediately when a page loads up, but can be triggered by the author's UI scripting. Finally, it sets the LOOP attribute to FALSE, assuming that the UI events to be sonified should only trigger one-shots that do not loop endlessly.

The following before-and-after example shows how preloadEmbed can be used in place of magicEmbed to simplify authoring. Apart from efficiency, the two statements are equivalent.

BEFORE (Using the magicEmbed method)
myMusicObject.magicEmbed (
   'SRC="trance.rmf" ' +
   'WIDTH=0 ' +
   'HEIGHT=0 ' +
   'HIDDEN ' + 
   'AUTOSTART=FALSE ' +
   'LOOP=FALSE'
);
AFTER (Using the preloadEmbed Method)
myMusicObject.preloadEmbed ('trance.rmf');

It's important to note that, while the magicEmbed method can take an EmbedLineSTR parameter, the preloadEmbed method does not accept this parameter type and can only accept a FileUrlSTR parameter. An optional additional ExtraEmbedAttributesSTR parameter lets you specify further embed attributes or override the default embed attributes that are set by this method.


  • preloadEmbed ( FileUrlSTR , ExtraEmbedAttributesSTR )

    This is a variation of the preloadEmbed method that takes an additional ExtraEmbedAttributesSTR string as a parameter. With this parameter, the author can either override the defaults that are set by preloadEmbed, or add additional attribute settings that are not automatically set by the preloadEmbed method.

    A case where you might want to use the optional parameter would be if you had to embed an RMF file for the sonification of a button click, but the file was a little too loud for the rest of the page's sonification. In this case, you could specify the additional attribute as follows:

    myMusicObject.preloadEmbed ('button-flourish.rmf','VOLUME=70');

With the new mechanism for embedding truly hidden player instances introduced in Music Object version 3.3, a problem was found with embedding hidden instances in table cells, that would cause intermittent crashes in some 4.x versions of Netscape Navigator. Hidden instances should, therefore, not be embedded within table cells (or any other container elements, for that matter). Hidden instances should only be embedded as top level members of the BODY tag. For more information, consult the section Authoring Issues: Embedding Hidden Instances in Table Cells.


  • It is only possible to embed one player instance per Music Object instance.
  • This method may only be called before the player instance has been embedded.

The stubEmbed method lets you preload a stub RMF file into a hidden player instance, specifically for the purpose of playing musical notes through that instance.

    stubEmbed ( FileUrlSTR )

The stubEmbed method is an authoring shortcut used when an author wishes to embed a hidden player for the purposes of sonifying the user interface of a page. Embedding of a stub file is a technique used often when sonifying the user's interactions on an HTML page with musical notes or Groovoids.

Essentially, the stubEmbed method calls the magicEmbed method with preset values for the WIDTH, HEIGHT, HIDDEN, AUTOSTART, and LOOP attributes. It is assumed that the author does not wish the UI sonification player to have a visible control panel, and so both the WIDTH and HEIGHT are set to 2 pixels, and the HIDDEN attribute is present. Also, it is assumed that the player instance is possibly to be used for note playing as well as Groovoid playing. Since notes can only be played to Beatnik while a music file is being played, the player is therefore embedded with the AUTOSTART and LOOP attributes both set to TRUE. This way, the player will always be in play mode and the author will be able to send playNote and noteOn messages.

The following before-and-after example shows how the embedding of a stub file can be simplified using the stubEmbed method rather than using the magicEmbed method. Apart from efficiency, the two statements are equivalent.

BEFORE (Using the magicEmbed method)
myMusicObject.magicEmbed (
   'SRC="stub.rmf" ' +
   'WIDTH=0 ' +
   'HEIGHT=0 ' +
   'HIDDEN ' +
   'AUTOSTART=TRUE ' +
   'LOOP=TRUE'
);
AFTER (Using the stubEmbed Method)
myMusicObject.stubEmbed ('stub.rmf');

It's important to note that, while the magicEmbed method can take an EmbedLineSTR parameter, the stubEmbed method does not accept this parameter type and can only accept a FileUrlSTR parameter. This method does also not accept the optional ExtraEmbedAttributesSTR parameter that the preloadEmbed method accepts.

With the new mechanism for embedding truly hidden player instances introduced in Music Object version 3.3, a problem was found with embedding hidden instances in table cells, that would cause intermittent crashes in some 4.x versions of Netscape Navigator. Hidden instances should, therefore, not be embedded within table cells (or any other container elements, for that matter). Hidden instances should only be embedded as top level members of the BODY tag. For more information, consult the section Authoring Issues: Embedding Hidden Instances in Table Cells.


  • If no stub file URL is specified, the default is 'stub.rmf' and so, if the file 'stub.rmf' is located in the same directory as the HTML page that you are authoring, it is not necessary to specify a stub file URL.
  • It is only possible to embed one player instance per Music Object instance.
  • This method may only be called before the player instance has been embedded.

The fadeVolume method lets you fade the overall playback volume of the Music Object instance from a specified starting volume to a specified ending volume over a specified amount of time.

    fadeVolume (
    StartVolume0to100 ,
    EndVolume0to100 ,
    FadeBOOLorFadeTimeMsINT ,
    FadeEndCallbackFunctionorJavaScriptSTR ,
    FadeStepCallbackFunctionorJavaScriptSTR
)

The fadeVolume method is a superset of the fadeTo method, but allows greater flexibility in control over the volume fade. In addition to specifying starting and ending volumes, you can also specify callbacks that are to be executed as part of the volume fade. The fade end callback lets you specify a callback that is to be executed when the fade has completed. And the fade advance callback lets you specify a callback to be executed for each time that the volume is adjusted during the course of the fade. Both these callbacks are helpful for synchronizing events within an HTML page to a fade that has been initiated.


  • When a value of null is specified for the StartVolumeLevel parameter, then the instance's current volume setting is used as the starting volume for the fade.
  • When a value of null is specified for the EndVolumeLevel parameter, then the fade will end at volume 0 (silence).
  • When a value of null is specified for the FadeBOOL or FadeTimeMsINT parameter, then the fade will have a default duration of 1 second (1000ms), which is equivalent to specifying true or 1000 as a value for this parameter.
  • When a value of null is specified for the FadeEndCallback parameter, no callback will be executed when the fade reaches its conclusion.
  • When a value of null is specified for the FadeStepCallback parameter, no callback will be executed each time the volume level is updated throughout the course of a fade.

  • fadeVolume ( StartVolume0to100 , EndVolume0to100 , FadeBOOLorFadeTimeMsINT , FadeEndCallbackFunctionorJavaScriptSTR )

    When no FadeStepCallback value is specified, no callback will be executed each time the volume level is updated throughout the course of a fade.

  • fadeVolume ( StartVolume0to100 , EndVolume0to100 , FadeBOOLorFadeTimeMsINT )

    When no FadeEndCallback value is specified, no callback will be executed when the fade reaches its conclusion.

  • fadeVolume ( StartVolume0to100 , EndVolume0to100 )

    When no FadeBOOL or FadeTimeMsINT parameter is specified, the fade will have a default duration of 1 second (1000ms), which is equivalent to specifying true or 1000 as a value for this parameter.

  • This method should only be called while the Music Object instance is playing.

The getFileSize method returns the size of the currently loaded file, measured in bytes.

Because the value returned by this method is measured in bytes, in order to calculate the KiloByte (Kb) size of the file, you would divide the value returned by 1024. function bytesToMb (bytes) {

}

The play method lets you start a file playing that is currently loaded into a Music Object instance, load or play a different music file than the file that is currently loaded by specifying a file URL, or resume playback of a Music Object instance that is in a paused state.

    play ( )

When used with no parameters, the play method will start the Music Object instance playing whichever file it currently has loaded. If the instance is already playing, then this method will have no affect and the instance will continue playing - the instance will not restart the file it is playing from the beginning. If a Music Object instance already has a file loaded, then to restart the file playing each time a certain event occurs, simply precede the play () instruction with a stop () instruction, as in myMusicObject.stop (); myMusicObject.play ();. If an instance is currently busy loading in a music file, then a play () instruction will have no affect. If the Music Object instance has been put into a paused state using the pause method, then a play () instruction will make the instance resume playing.

The setTransposable method lets you set whether or not the specified MIDI channel in an RMF or MIDI file is transposable.

    setTransposable ( MidiChannel0to16 , TransposableBOOL )

By default, all MIDI channels excepting the percussion channel (channel 10) are transposable, unless set otherwise using this method. If a MIDI channel is transposable, then the musical notes playing on that channel will be affected / offset by the transposition setting for the Music Object instance (as set using the setTranspose method).

On the other hand, if a MIDI channel is not transposable, then the transposition setting for the instance will have no affect on the musical notes that play through that channel. This is particularly useful for RMF files that contain MIDI channels that are playing custom instrument drum loops. If such channels were to be transposed along with the other channels in the music file, then the tempo of the drum loops would be affected as the drum loop samples are pitch-shifted, creating undesirable chaos in the overall rhythm of the music.

Another useful application for the non-transposable state is the case where a MIDI channel is playing percussion instruments in pitched mode using NRPN640 (see the Beatnik Editor documentation) to achieve "talking drum" effects. It is most likely that such percussion effects will not be in the specific key of the melodic elements of the musical arrangement, and transposing them would produce inappropriate results. By setting such MIDI channels to be non-transposable, problems like this can be avoided when transposing the music using the setTranspose method.

You can toggle the transposable state for a specific MIDI channel by using the getTransposable method in conjunction with the setTransposable method, as in the example myMusicObject.setTransposable (!myMusicObject.getTransposable (13));.


  • This method should only be called while the Music Object instance is playing.

The following custom function takes a Music Object instance as a parameter and returns a boolean value indicating the instance's stopped state...

function isStopped (musicObjectRef) {
   return !musicObjectRef.isPlaying () && !musicObjectRef.isPaused ();
}

One would call the above function passing the Music Object instance as a parameter, as in the statement if (isStopped (myMusicObject)) {updateButtonsDisplay ()}.

 File Types Supported   RMF, MIDI 
 

The onMetaEvent method lets you register the function to be executed each time the Music Object instance encounters a meta event while playing an RMF or MIDI file.

    onMetaEvent ( HandlerFunction )

Handler Must be a Function

Unlike other methods that register callback handlers, the onMetaEvent method only accepts a Function parameter type for the callback handler. The function that is registered as a handler for meta event callbacks must expect to receive two string parameters. The first parameter will contain a value of type MetaEventTypeSTR, indicating the type of meta event that has been encountered. The second parameter will contain a string value, indicating the value of the meta event that has been encountered. For instance, if a marker named "Chorus Starts" were encountered, then the first parameter would contain the string value "Marker" and the second parameter would contain the string value "Chorus Starts".

The possible values for the first parameter are:

  • GenericText
  • Copyright
  • Lyric
  • Marker
  • CuePoint
  • Unknown

TrackName events are not currently supported by the Beatnik Player, since it was decided that the sheer volume of these events clumped up at the beginning of RMF and MIDI files may produce too much load on the JavaScript interpreter and may "bring down" the JavaScript code intended to handle meta events.

Define the Handler Function First

It's very important that the function you wish to register as a handler for meta event callbacks be defined before you call the onMetaEvent method. If your handler function is not defined beforehand then your code will produce JavaScript errors and will not work correctly. For example, compare the following two code segments...

  INCORRECT
myMusicObject = new Music ();
myMusicObject.onMetaEvent (myMetaEventHandler);

function myMetaEventHandler (metaEventType,metaEventValue) {
   // some code to handle meta events
}
  CORRECT
function myMetaEventHandler (metaEventType,metaEventValue) {
   // some code to handle meta events
}

myMusicObject = new Music ();
myMusicObject.onMetaEvent (myMetaEventHandler);

An Example

In the following hypothetical example, we're assuming that the Music Object instance "myMusicObject" is playing a music file that contains marker events to indicate when the chorus sections start and end. The function myMetaEventHandler is registered as the meta event handler for this instance, using the statement myMusicObject.onMetaEvent (myMetaEventHandler).

//...........................
// more JavaScript code above

function myMetaEventHandler (metaEventType,metaEventValue) {
   if (metaEventType == 'Marker') {
      if (metaEventValue == 'Chorus Starts') {
         // code to SHOW chorus lyrics layer
      } else if (metaEventValue == 'Chorus Ends') {
         // code to HIDE chorus lyrics layer
      }
   }
}

myMusicObject = new Music ();
myMusicObject.onMetaEvent (myMetaEventHandler);

// more JavaScript code below
//...........................

The function myMetaEventHandler contains conditional code to check for the markers "Chorus Starts" and "Chorus Ends" and to then update the display of the HTML page appropriately. The function myMetaEventHandler may be called many times while the music file is playing for other meta events that are encountered, but because of the conditional "if" statements, the function will only ever do something when a marker event is encountered that has either the value "Chorus Starts" or "Chorus Ends".

The specific code that you would place inside the meta event handler function will depend entirely on the types of events you wish to filter out and act upon. If you wanted, you could write your handler function so that it always does the same thing, regardless of the type or value of meta events encountered. Another possibility would be to write a handler function that always does the same thing, but for only a specific type of meta event. It all depends on the goals of your project.
